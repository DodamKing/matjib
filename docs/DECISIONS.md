# 설계 결정 기록 (ADR-lite)

새 결정이 생기면 맨 아래에 추가. "왜"를 남겨서 재토론(=토큰 낭비)을 막는 게 목적.

---

## D1. 기술 스택: Next.js + TypeScript (풀스택)
- **결정**: 별도 백엔드 없이 Next.js App Router 한 프로젝트로.
- **왜**: 원클릭 웹 진입 = 웹앱 본질에 맞음. API Route로 키를 숨긴 채 외부 API 호출 가능.
  현재 규모(API 호출 몇 개)에 FastAPI 등 분리 백엔드는 과함.

## D2. 1차 데이터 소스: 소상공인시장진흥공단 상가(상권)정보 API
- **결정**: 공공데이터포털의 「반경내 상가업소 조회」를 1차 소스로 사용.
- **검증됨 (2026-06-19)**:
  - 요청: `radius`(m) + `cx`(경도) + `cy`(위도) + `indsLclsCd`(업종 대분류)
  - 응답: 상호명, 상권업종(대/중/소분류), 경도/위도, 주소
  - 업종 분류: 대분류 10 / 중분류 75 / 소분류 247. 음식 대분류로 식당 필터, 소분류로 면/국밥 등 세분 매칭.
  - 무료. data.go.kr 가입 + API 신청 → 보통 1~2시간 내 키 발급.
- **출처**: https://www.data.go.kr/data/15012005/openapi.do

## D3. ⚠️ 데이터 함정: 품질/평점 신호 없음 → "최적"의 정의를 바꾼다
- **사실**: 위 데이터는 국세청·카드사 기반 "등록 정보". 리뷰·평점·맛 품질 신호가 **전혀 없음**. 분기 갱신.
- **결정**: "광고 제외 최적의 3개"에서 *최적*을 품질로 정하지 않는다.
  정렬/선정 = **거리 + 업종 적합도 + 공정 셔플(랜덤 추첨)**.
- **왜 괜찮은가**: 기획서 USP(차별점 3 "평점 조작 없는 플랫·공정 데이터")와 오히려 일치.
  컨셉을 "AI가 맛집을 고른다" → "AI가 상황을 진단해 가까운 곳을 공정하게 처방한다"로 정렬.

## D4. 상호명+좌표 보강 = 지도/길찾기 링크용 (평점 랭킹 비목표)
- **결정**: 공공데이터로 후보(좌표·거리·업종) 확보 → 상호명+좌표로 카카오 로컬 API 재조회해
  **길찾기/상세 링크·정확 주소·전화**를 보강. **선택적(fallback)**: 매칭 실패 시 공공데이터만으로 카드 표시.
- **하지 않는 것**: 평점/리뷰 수집(스크래핑 포함). USP와 충돌하고 ToS 위험. 퍼지 매칭이라 품질 근거로 부적합.

## D5. AI 매칭 = 자연어 → 업종 "번역기"
- **결정**: Claude API로 "비 오는데 뜨끈한 면" → 업종 소분류 코드 + 키워드로 변환. 맛 판단은 안 함.
- **모델**: 분류/번역은 가볍고 빠른 작업 → `claude-haiku-4-5` 기본. 품질 부족하면 `claude-sonnet-4-6`로 상향.
- **페르소나**: 결과 카피("AI 의사 처방" 톤)는 별도 프롬프트로. 데이터 정확성과 분리.

## D6. API 키는 서버에서만
- **결정**: 공공데이터/Anthropic/카카오 키는 API Route(서버)에서만 사용. 클라이언트 노출 금지.
